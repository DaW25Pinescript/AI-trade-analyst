**[CRITICAL-1] ExecutionRouter._run_arbiter_and_finalise — macro_context and overlay_delta_reports silently dropped**

**Type:** Confirmed Issue
**Location:** `ai_analyst/core/execution_router.py:L200-204`

```python
arbiter_prompt = build_arbiter_prompt(
    analyst_outputs=all_outputs,
    risk_constraints=self.ground_truth.risk_constraints,
    run_id=self.run_id,
    # ← macro_context MISSING
    # ← overlay_delta_reports MISSING
)
```

**Why this is a problem:**
`build_arbiter_prompt()` accepts `macro_context` and `overlay_delta_reports` as keyword arguments. When running via CLI (`python cli.py run ...`), hybrid mode, or manual mode, the arbiter call goes through `ExecutionRouter._run_arbiter_and_finalise()`. Neither parameter is passed. The macro context injected by `macro_context_node` during the graph run is never seen by the arbiter in these modes, and overlay delta evidence is discarded even when overlay charts were submitted.

The FastAPI path (`graph/arbiter_node.py`) does pass both. So the arbiter behaves **differently depending on entry point** — a fundamental correctness split.

**Real-world impact:**
A hybrid-mode or CLI user who submitted a 15M overlay chart gets an arbiter verdict that ignores that overlay data entirely. MRO macro context is also silently absent from the prompt even when `macro_context_node` successfully fetched it.

**Recommended fix:**
Pass both arguments. The `ExecutionRouter` already has access to `self.ground_truth` (which carries `m15_overlay`). The macro context needs to be fetched separately or passed in at construction time:

```python
# In ExecutionRouter.__init__:
self.macro_context: Optional[MacroContext] = None  # populated by caller or via scheduler

# In _run_arbiter_and_finalise:
overlay_delta_reports = self._loaded_overlay_deltas or []
arbiter_prompt = build_arbiter_prompt(
    analyst_outputs=all_outputs,
    risk_constraints=self.ground_truth.risk_constraints,
    run_id=self.run_id,
    overlay_delta_reports=overlay_delta_reports,
    overlay_was_provided=bool(self.ground_truth.m15_overlay),
    macro_context=self.macro_context,
)
```

**[CRITICAL-2] Synchronous HTTP in async pipeline blocks the event loop (MRO ingestion)**

**Type:** Confirmed Issue
**Location:** `macro_risk_officer/ingestion/clients/finnhub_client.py:L79`, `fred_client.py:L55`, `gdelt_client.py:L65`

```python
# finnhub_client.py:79 — called from async macro_context_node
response = httpx.get(
    f"{_BASE_URL}/calendar/economic",
    ...
    timeout=10.0,
)
```

**Why this is a problem:**
`MacroScheduler._refresh()` calls all three clients synchronously. `_refresh()` is called by `get_context()`, which is called from the async `macro_context_node`. Synchronous `httpx.get()` blocks the asyncio event loop for up to 10 seconds per provider (3 providers × 10s = up to 30s stall on cold start). Under load this starves other concurrent requests.

**Real-world impact:**
Under production concurrent load, the first TTL miss after server startup can stall all in-flight `/analyse` requests for up to 30 seconds.

**Recommended fix:**
Convert clients to async httpx and make `_refresh` an async function:

```python
# finnhub_client.py
async def fetch_calendar(self, ...) -> List[MacroEvent]:
    async with httpx.AsyncClient(timeout=10.0) as client:
        response = await client.get(...)

# scheduler.py — use asyncio.to_thread as a bridge until full async migration
import asyncio
ctx = await asyncio.to_thread(self._refresh, instrument)
```

**[CRITICAL-4] Overlay delta node assigns wrong model config to surviving analysts after Phase 1 partial failure**

**Type:** Confirmed Issue
**Location:** `ai_analyst/graph/analyst_nodes.py:L155-158`

```python
tasks = [
    run_overlay_delta(
        ANALYST_CONFIGS[i % len(ANALYST_CONFIGS)],  # ← re-indexes by position, not by original slot
        build_overlay_delta_prompt(ground_truth, analyst_output),
        ground_truth.run_id,
    )
    for i, analyst_output in enumerate(analyst_outputs)
]
```

`analyst_outputs` is assembled from `valid_outputs` (only passing analysts, in result order). If `ANALYST_CONFIGS[0]` (gpt-4o) and `ANALYST_CONFIGS[2]` (gemini) succeed but `[1]` and `[3]` fail, `valid_outputs = [output_from_gpt4o, output_from_gemini]`. The overlay loop then calls `ANALYST_CONFIGS[0]` for output 0 (correct) and `ANALYST_CONFIGS[1]` (gpt-4o, wrong model) for output 1 (should be gemini). The gemini Phase 1 baseline is compared against a gpt-4o Phase 2 call — persona mismatch and cross-contamination.

**Real-world impact:**
Under any partial failure (very common when Grok or Gemini is unavailable), overlay delta analysis uses the wrong model–persona combination, compromising the isolation guarantee that is a core design property.

**Recommended fix:**
Track the original config alongside the output:

```python
# In parallel_analyst_node — collect (config, output) pairs:
valid_pairs: list[tuple[dict, AnalystOutput]] = []
for i, result in enumerate(results):
    if isinstance(result, AnalystOutput):
        valid_pairs.append((ANALYST_CONFIGS[i], result))

state["analyst_outputs"] = [o for _, o in valid_pairs]
state["analyst_configs_used"] = [c for c, _ in valid_pairs]  # add to GraphState

# In overlay_delta_node:
for config, analyst_output in zip(state["analyst_configs_used"], analyst_outputs):
    run_overlay_delta(config, build_overlay_delta_prompt(...), ...)
```

**[HIGH-1] Retry logic retries ALL exceptions, including non-retriable ones (e.g., AuthenticationError, ValidationError)**

**Type:** Confirmed Issue
**Location:** `ai_analyst/core/llm_client.py:L30`

```python
except Exception as exc:  # retries ValueError, AuthenticationError, everything
    last_error = exc
    if attempt >= attempts:
        break
    await asyncio.sleep(retry_backoff_s * attempt)
```

Linear backoff of 0.4 s and 0.8 s is far too short for rate limits (429 typically requires 1–60 s). Auth errors (401/403) should never be retried at all.

**Recommended fix:**
```python
from litellm.exceptions import AuthenticationError, RateLimitError

RETRIABLE = (asyncio.TimeoutError, RateLimitError, httpx.ConnectError)
NON_RETRIABLE = (AuthenticationError, ValidationError)

except Exception as exc:
    if isinstance(exc, NON_RETRIABLE):
        raise  # never retry auth failures
    last_error = exc
    if attempt >= attempts:
        break
    # Exponential backoff with jitter for rate limits
    backoff = retry_backoff_s * (2 ** (attempt - 1)) + random.uniform(0, 0.5)
    await asyncio.sleep(min(backoff, 60.0))
```

**[HIGH-4] MacroScheduler is not thread-safe — thundering herd on cache miss**

**Type:** Architectural Risk
**Location:** `macro_risk_officer/ingestion/scheduler.py:L66-84`

```python
def get_context(self, instrument: str = "XAUUSD") -> Optional[MacroContext]:
    now = time.monotonic()
    if self._cache is not None and (now - self._last_fetch) < self.ttl:  # ← no lock

    def __init__(self, ...):
        self._lock = threading.Lock()  # or asyncio.Lock() for pure-async

    def get_context(self, instrument):
        with self._lock:
            # ... existing logic
```

**[MED-1] FRED event timestamps anchored to first of month — introduces up to 28-day decay error**

**Type:** Confirmed Issue
**Location:** `macro_risk_officer/ingestion/clients/fred_client.py:L119`

```python
timestamp=datetime(now.year, now.month, 1, tzinfo=timezone.utc),  # monthly release anchor
```

DFF released on March 28 is timestamped March 1. `DecayManager` computes `age_hours` from this timestamp, making the event appear 27 days fresher than it is. For tier-1 DFF with a 168-hour half-life, this means an event that should decay to `exp(-648/168) ≈ 0.02` appears as `exp(-648/168)` after days, but the inflated freshness is fed to the reasoning engine.

**Recommended fix:** Use the actual FRED observation date for the timestamp:
```python
obs = self.fetch_latest(series_id, n_obs=2)
# obs[0]["date"] is the actual observation date, e.g. "2026-02-01"
actual_ts = datetime.fromisoformat(obs[0]["date"]).replace(tzinfo=timezone.utc)
```

**[MED-2] GDELT geopolitical events use artificial actual/forecast values — amplifies surprise calculation**

**Type:** Confirmed Issue
**Location:** `macro_risk_officer/ingestion/clients/gdelt_client.py:L106-112`

```python
if avg_tone < _ESCALATION_THRESHOLD:
    actual = 1.0
    forecast = 0.0  # surprise = 1.0 / 0.0 → ZeroDivisionError branch → magnitude = abs(1.0)
```

With `forecast=0`, the `_surprise_multiplier` in `reasoning_engine.py:L158` hits the `event.forecast == 0` guard and returns `1.0`. But this bypasses the actual surprise magnitude entirely — every GDELT event always produces the same tier-2 contribution regardless of how extreme the tone signal was.

Additionally, the tier is always 2, meaning major geopolitical events (war declarations, OPEC cuts) are never tier-1 escalated.

**Recommended fix:** Use `avg_tone` magnitude as a proxy for surprise intensity, and allow tier-1 for extreme tone deviations:
```python
actual = avg_tone / 20.0   # normalize to approx [-1, 1]
forecast = 0.0
tier = 1 if abs(avg_tone) > 10.0 else 2
```

## TASK 4 — Summary & Roadmap

### 1) Health Score: **Good**

The AI Trade Analyst codebase demonstrates unusually strong architectural discipline for a project of this scale and complexity. The schema engineering (frozen GroundTruthPacket, typed ScreenshotMetadata, complete migration chain, prompt-level design rule enforcement), the two-phase clean/overlay isolation, and the fail-silent MRO integration are all production-quality decisions that are rarely this well-executed. The 514-test baseline, modular graph/core/models separation, and release infrastructure (Docker + Compose + CI) signal a team that ships with discipline.

However, three confirmed issues rise to critical before any production deployment: the ExecutionRouter silently drops macro context in all non-API paths (CRITICAL-1), the browser bridge will always timeout on multi-model analysis (CRITICAL-3), and the overlay delta phase can assign the wrong model to the wrong analyst after partial Phase 1 failures (CRITICAL-4). Alongside these, the synchronous HTTP blocking in the MRO ingestion layer and the non-existent Grok model name are active regressions affecting every run. The technical debt register is manageable but the browser ↔ Python schema divergence is a compounding cross-cutting concern that will become progressively harder to address.

---

### 2) Top 5 Actionable Items

**#1 — Fix the ExecutionRouter macro_context + overlay_delta omission (CRITICAL-1)**

*Why it ranks here:* This is a silent correctness bug on every CLI/hybrid/manual run. It means the MRO P1–P4 work is functionally absent from those paths. It is also the easiest fix on this list.
*Expected payoff:* Macro context and overlay data reach the arbiter in all execution modes — the pipeline is correct everywhere, not just via FastAPI.
*Effort:* 1–2 hours.

---

**#2 — Raise the browser bridge timeout to 3 minutes (CRITICAL-3)**

*Why it ranks here:* The G11 API bridge feature is functionally broken for all users today. Every automated analysis attempt from the browser will appear to fail after 12 seconds.
*Expected payoff:* G11 bridge becomes usable immediately.
*Effort:* 30 minutes (code change + test).

---

**#3 — Convert MRO ingestion clients to async httpx (CRITICAL-2)**

*Why it ranks here:* Blocking the event loop for 10–30 s on TTL miss degrades all concurrent requests. With multi-worker deployment this is a production reliability issue.
*Expected payoff:* MRO fetch on cache miss becomes non-blocking; server concurrency restored.
*Effort:* M — 2–4 hours across 3 client files + scheduler + tests.

---

**#4 — Fix overlay delta model alignment + add integration test (CRITICAL-4)**

*Why it ranks here:* The isolated-context guarantee is a core design property. Silently violating it on any partial failure undermines a key architectural trust boundary.
*Expected payoff:* Phase 2 correctness restored; the design contract is testable.
*Effort:* M — 3–4 hours including GraphState extension and test.

---

**#5 — Add image size validation + per-run cost ceiling (HIGH-6)**

*Why it ranks here:* Without it, a single request with oversized charts can produce $10–$20+ API charges. As the tool is shared or opened to network access, this becomes an active financial risk.
*Expected payoff:* Eliminates runaway cost scenarios; adds a hard safety boundary.
*Effort:* S — 2–3 hours for validation + env var guard + test.

### 3) Long-Term Architectural Recommendations

**G12 — Complete the overlay metadata capture in the browser UI**

The `backup_validation.js` guard that forces `m15Overlay: null` needs to be lifted. G12 should define the full typed overlay metadata schema in the browser's backup_validation.js and update `storage_local.js` / `model.js` to capture `{timeframe, lens, evidenceType, indicatorClaims, indicatorSource, settingsLocked}` from the form. This unblocks export/import of overlay-enabled tickets and enables future multi-run overlay comparison.

**Usage Observability — Centralize run telemetry into a queryable store**

The per-run `usage.jsonl` approach works well for isolation but makes cross-run aggregation (total monthly cost, model performance comparison, analyst failure rate by provider) difficult. Consider a lightweight SQLite `usage_db.py` that mirrors the JSONL schema into a queryable table — the `FetchLog` in `metrics.py` is already the right pattern. The existing `summarize_usage()` function can be extended to read from either source.

**Token/Cost/Run Telemetry — Expose metrics endpoint**

Add a `GET /metrics` endpoint (or Prometheus-compatible `GET /metrics/prometheus`) that exposes:
- `total_runs_today`, `total_cost_usd_today`
- `model_call_success_rate` by provider
- `macro_context_availability_pct` (from FetchLog KpiReport)
- `avg_pipeline_latency_ms`

This enables deployment-time visibility without requiring log aggregation infrastructure.

**Browser ↔ Backend Contract Hardening — Shared JSON Schema**

The most impactful long-term investment is defining a canonical `analyst_output.schema.json` and `final_verdict.schema.json` in `docs/schema/` that is the single source of truth for both `analyst_prompt_builder.py` (OUTPUT_SCHEMA string) and `promptGenerator.js` (outputShape string), and validated by both `AnalystOutput` (Python) and any future JS import of AI responses. This eliminates the current divergence (JS uses 0–100 confidence, Long/Short/Wait; Python uses 0.0–1.0, LONG/SHORT/NO_TRADE/WAIT).

**Prompt System Unification**

The v1.1 and v1.2 directories should be collapsed to a single active version with an explicit `PROMPT_LIBRARY_VERSION` env var or config key. The lens files in `chart_analysis/` and `prompt_library/v1.2/` serve overlapping purposes and should be consolidated with a clear precedence rule. Long-term, consider a prompt manifest YAML that documents each lens file's contract, version, and activation conditions.

**Schema Evolution Strategy**

The migration chain in `migrations.js` is clean but purely append-only. As the schema complexity grows (G12, releases, telemetry fields), consider a contract-based migration approach:
1. Every schema bump requires a migration function + a round-trip test in `test_schema_round_trip.py` (Python) and `test_migrations.js` (JS).
2. `schemaVersion` should be treated as a feature flag: code reads `schemaVersion` at load time and activates/deactivates field handling accordingly.
3. A `schema_changelog.md` in `docs/schema/` should document the breaking vs additive change for each version.

**LangGraph / FastAPI / MRO Boundary**

The MRO scheduler is currently a module-level singleton initialized lazily on first request. For long-lived FastAPI deployments:
1. Move scheduler initialization to `@app.on_event("startup")` and attach to `app.state.mro_scheduler`.
2. Pass the scheduler instance into `macro_context_node` via the initial GraphState (or FastAPI's DI system) rather than relying on a module global.
3. This enables per-test scheduler injection, clean shutdown via `@app.on_event("shutdown")`, and multi-worker-safe instantiation.

**Scaling the multi-model fan-out**

When all four analysts are running against vision models with 45 s timeouts, the parallel fan-out can consume all uvicorn worker threads simultaneously. Consider:
1. A dedicated thread pool or asyncio semaphore limiting concurrent LLM calls: `async with asyncio.Semaphore(6): await acompletion_metered(...)`.
2. A background task queue (Celery, ARQ) for long-running analysis jobs so `/analyse` returns a `run_id` immediately and a `GET /runs/{run_id}/result` polls for completion — especially important once Phase 2 overlay analysis adds more parallel calls.
